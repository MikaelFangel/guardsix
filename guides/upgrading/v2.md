# Upgrading to v2

v2 is a full redesign of the library. This guide covers every breaking change
and how to migrate your code.

## Dependency

```elixir
# v1
{:guardsix, "~> 1.0.0"}

# v2
{:guardsix, "~> 2.0.0"}
```

The HTTP client changed from HTTPoison to Req, and Joken was added for
JWT-authenticated endpoints (alert rules, repos).

## Client setup

The plain credentials map is replaced by a `Client` struct built via the
facade.

```elixir
# v1
credentials = %{
  ip: "192.168.1.10",
  username: "admin",
  secret_key: "mysecret",
  verify_ssl: false
}

# v2
client = Guardsix.client("https://192.168.1.10", "admin", "mysecret", ssl_verify: false)
```

Key differences:

- `ip` is now a full URL (`base_url`)
- `verify_ssl` is renamed to `ssl_verify`
- SSL verification defaults to `true` (was `false` in v1)

## Search

Guardsix.Query is replaced by `Guardsix.search_params/4,5`.
`run_search` is back with the same polling behaviour as v1 — see the
[Polling for Search Results](polling-search-results.md) guide for details.

```elixir
# v1
query = %Guardsix.Query{
  query: "user=*",
  time_range: [1_714_986_600, 1_715_031_000],
  limit: 100,
  repos: ["127.0.0.1:5504"]
}
{:ok, result} = Guardsix.run_search(credentials, query)

# v2
query = Guardsix.search_params("user=*", 1_714_986_600, 1_715_031_000, 100, ["127.0.0.1:5504"])
{:ok, result} = Guardsix.run_search(client, query)
```

Option names changed: `:poll_interval` → `:polling_interval`, `:max_retries` → `:max_attempts`.
The timeout error is now `{:error, :max_attempts_exceeded}` (atom instead of string).

| v1 | v2 |
|---|---|
| Guardsix.run_search/2 | `Guardsix.run_search/3` |
| Guardsix.get_search_id/2 | `Search.get_id/2` |
| Guardsix.get_search_result/2 | `Search.get_result/2` |
| Guardsix.user_preference/1 | `Search.user_preference/1` |
| Guardsix.guardsix_repos/1 | `Search.guardsix_repos/1` |
| Guardsix.devices/1 | `Search.devices/1` |
| Guardsix.livesearches/1 | `Search.livesearches/1` |

## Incidents

All incident functions moved to `Guardsix.Core.Incident`.

```elixir
# v1
{:ok, incidents} = Guardsix.incidents(credentials, start_time, end_time)
{:ok, _} = Guardsix.resolve_incidents(credentials, ["id1"])
{:ok, _} = Guardsix.add_comments(credentials, %{"id1" => ["note"]})

# v2
alias Guardsix.Core.Incident

{:ok, incidents} = Incident.list(client, start_time, end_time)
{:ok, _} = Incident.resolve(client, ["id1"])

comment = Guardsix.comment("id1", "note")
{:ok, _} = Incident.add_comments(client, [comment])
```

| v1 | v2 |
|---|---|
| Guardsix.incidents/3 | `Incident.list/3` |
| Guardsix.incident_states/3 | `Incident.list_states/3` |
| Guardsix.incident/3 | `Incident.get/3` |
| Guardsix.add_comments/2 | `Incident.add_comments/2` |
| Guardsix.assign_incidents/3 | `Incident.assign/3` |
| Guardsix.resolve_incidents/2 | `Incident.resolve/2` |
| Guardsix.close_incidents/2 | `Incident.close/2` |
| Guardsix.reopen_incidents/2 | `Incident.reopen/2` |
| Guardsix.users/1 | `Incident.get_users/1` |

New in v2: `list/4` and `list_states/4` accept an optional filters map.

## Removed modules

| Removed | v2 replacement |
|---|---|
| Guardsix.Query | `Guardsix.search_params/4,5` |
| Guardsix.Incident | `Incident.get/3` |
| Guardsix.TimeRange | Inline args |
| Guardsix.IncidentComment | `Guardsix.comment/2` |
| Guardsix.IncidentCommentData | Pass `[Comment.t()]` directly |
| Guardsix.IncidentIDs | Pass `[String.t()]` directly |

## New in v2

These domains are entirely new and have no v1 equivalent:

- **Alert Rules** — `Guardsix.Core.AlertRule` with builder pattern via `Guardsix.rule/1`
- **Email Notifications** — builder via `Guardsix.email_notification/2`
- **HTTP Notifications** — builder via `Guardsix.http_notification/3`
- **Guardsix Repos** — `Guardsix.Core.GuardsixRepo`
- **User-Defined Lists** — `Guardsix.Core.UserDefinedList`
- **Incident filtering** — optional filters map on `list/4` and `list_states/4`

See the `Guardsix` module docs for full usage examples.
